  • リポジトリにコミットしてプッシュする。

bombay-vim

かの有名なテキストエディタ、 Vim の自作カラーテーマ。

  • 標準にあるカラーテーマ desert が好きだったが見づらいので自作。
  • 若干視認性があがった。
  • モチベーションを上げるための道具磨き大事。

perceptron-go

単純だけどそこそこの性能を出せる、学習アルゴリズムの一つ。

  • データがあれば、メールのスパムフィルターみたいな分類器をつくれる。
  • 動くけど、まだ実際に使うには未実装部分多い、テストコードない。
  • Golang で実装。

Kompl

文脈に合わせて自然な単語をサジェストする機能を提供するサーバー。

  • Vim から neocomplete 経由で叩くために実装中。
  • 直前の単語 N-gram + 入力中の単語の接頭辞から予測。
  • 複数候補サジェストされ、より自然なものほど上位にくる。
  • メモリと速度との戦い。
  • これも Golang で実装。

なぜ緑化活動するのか

1. 蓄積して再利用するためにやる

今まで

  • やっつけで書いたコードを残す習慣がなかった。
  • きれいなコードじゃないと人には見せたくない。
  • いらないと思っても、案外後で使う場面がでてくる。

これから

  • ゴミみたいなコードでも極力残す、公開する。
  • というかそもそも案外コード読む人いない。
  • 利用頻度が高ければ、ちょっとずつ修正が入って綺麗になる。

2. 継続する習慣をつけるためにやる

何かつくろうと思っても、大体挫折してきたので、やり方を変えてみた。

  • なるべく活動を GitHub に残して、なんで挫折したのか振り返る。
  • 質は気にせず、まずは時間を投入して量をだす。

やってみてわかったのは、

  • やってみたら意外と続けるだけなら簡単。
  • 一応動く状態にはやく持っていくとモチベーションがあがる。
  • タスクを細かくして、長丁場をつくらない。長いと辛い。

3. 履歴書のようなものだからやる

  • コードはエンジニアにとっての履歴書。
  • しょぼくても何かやってると案外声がかかる。
    • 大体スパムだけど、言語処理関係の OSS のおさそいなど。
  • 自分の技術レベルが良くも悪くも GitHub に反映されるようにしたい。

4. 自己満足感を得るためにやる
